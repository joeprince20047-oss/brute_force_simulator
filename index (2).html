Attack Modes

Three real-world
techniques, one tool.

Each mode mirrors how real attackers operate, so defenders can understand the threat and build better countermeasures.

📖
Dictionary Attack

Iterates through a pre-built wordlist of common passwords. Memory-efficient generator yields one password at a time, supporting wordlists of any size without loading them fully into RAM.

--mode dictionary
⚙️
Pure Brute Force

Exhaustively generates every character combination up to a configurable maximum length using itertools.product. Demonstrates why short passwords are trivially cracked.

--mode bruteforce
🗂️
Credential Stuffing

Reads user:pass pairs from a credential dump file, testing each pair exactly. Simulates how leaked databases from one breach are used against other services.

--mode stuffing
Architecture

Modular. Testable.
Clean separation.

Every component has a single responsibility and its own test file. Swap out any module without touching the rest.

⚡
Attack Engine
src/engine/attacker.py
Thread-safe queue, worker pool, HTTP POST requests, response classification, rate-limit detection and back-off.
📋
Wordlist Loader
src/engine/wordlist.py
Generator-based loading for memory efficiency. Supports dictionary files, brute force combination generation, and credential pair parsing.
📊
Reporter
src/engine/reporter.py
Thread-safe stat tracking using locks. Saves discovered credentials to timestamped CSV and JSON files in logs/.
🎯
Sandbox Server
src/target/server.py
Flask login endpoint running on localhost only. Includes account lockout (5 attempts → 30s cooldown) and HTTP 429 rate-limiting.
🖥️
GUI Dashboard
src/gui/dashboard.py
Tkinter real-time stats viewer. Shows attempts/sec, success count, and found credentials, updating every 500ms using after() callbacks.
⚙️
Config
src/engine/config.py
Validated Python dataclass holding all runtime settings. Raises ValueError on invalid inputs before the attack starts.

Quick Start

Up and running
in four steps.

The simulator and sandbox server both run locally. No external services needed.

01
Clone & install dependencies
Clone the repo and install all Python packages from requirements.txt.
git clone https://github.com/YOUR_USERNAME/brute-force-simulator.git cd brute-force-simulator pip install -r requirements.txt
02
Start the sandbox target server
Launch the local Flask login server. This is the only safe, ethical target for the simulator.
python -m src.target.server # → Running at http://127.0.0.1:5000/login
03
Run the simulator
Open a second terminal and launch the attack against the sandbox. Use --gui for the live dashboard.
python main.py \ --target http://127.0.0.1:5000/login \ --wordlist wordlists/common_passwords.txt \ --threads 5 # With live GUI dashboard: python main.py --target http://127.0.0.1:5000/login \ --wordlist wordlists/common_passwords.txt --gui # Dry run (no real requests): python main.py --target http://127.0.0.1:5000/login \ --wordlist wordlists/common_passwords.txt --dry-run
04
Run the test suite
23 unit tests across engine, wordlist, and auth modules. GitHub Actions runs these automatically on every push.
pytest tests/ -v
Ethical Guidelines

Built responsibly.
Use responsibly.

Understanding how attacks work is how defenders build better systems. That's the only purpose of this tool.

⚠️ Ethical Use Requirements

  • Only run the simulator against the included localhost Flask sandbox server or systems you explicitly own.
  • Never test against real websites, cloud services, APIs, or any system without written permission from the owner.
  • Do not include real leaked password databases in any public GitHub repository.
  • Use the --dry-run flag to preview what would be attempted before sending real requests.
  • This project is for educational demonstration of attack techniques and their defences — not for unauthorized access.
  • Present both the attacker's perspective and the defender's countermeasures (lockout, rate limiting) in any demonstration.
